# TODO

1. ✅ Add support for `not_null` assertion. Parameters: `path`. Validates that the value at the given path is not null.
2. ✅ Add support for `response_time` assertion. Parameters: `milliseconds` (number). Validates that the API response was received within the specified number of milliseconds.
3. ✅ Add support for `not_empty` assertion. Parameters: `path`. Similar to `not_null` but additionally validates that if the value at the path is a string, it cannot be an empty string.
4. ✅ Add support for `is_null` assertion. Parameters: `path`. Validates that the property at the given path either does not exist or has a null value.
5. ✅ Add support for `range` assertion. Parameters: `path`, `min`, `max`. Validates that the numeric value at the given path is between `min` and `max` (inclusive). If the value is a string, attempt to parse it as a number before comparing. Fail the assertion if the value is neither a number nor a parseable string, or if parsing fails.
6. ✅ Add support for `assert_true` assertion. Parameters: `path`. Validates that the value at the given path is boolean `true`. Any other value (including the string `"true"`, `1`, `null`, etc.) is a failed assertion.
7. ✅ Add support for `assert_false` assertion. Parameters: `path`. Validates that the value at the given path is boolean `false`. Any other value (including the string `"false"`, `0`, `null`, etc.) is a failed assertion.
8. ✅ Add support for `one_of` assertion. Parameters: `path`, `expected` (array of scalar values — strings, numbers, or booleans). Validates that the value at the given path is equal to one of the items in the `expected` array. Fails the assertion if the actual value is not in the array. Throws an error at evaluation time if any element of the `expected` array is a non-scalar type (e.g. object or array).
9. ✅ Add support for `regex_match` assertion. Parameters: `path`, `expected` (a regex pattern string). Validates that the value at the given path is a string and that the regex pattern in `expected` matches it. Fails the assertion if the value is not a string or if the regex does not match.
10. Add support for `starts_with` assertion. Parameters: `path`, `expected` (string). Validates that the string value at the given path starts with the expected prefix. Fails if the value is not a string or does not start with the prefix.
11. Add support for `ends_with` assertion. Parameters: `path`, `expected` (string). Validates that the string value at the given path ends with the expected suffix. Fails if the value is not a string or does not end with the suffix.
12. Add support for `status_in` assertion. Parameters: `expected` (array of integers). Validates that the HTTP response status code is one of the values in the `expected` array. Useful when multiple status codes are acceptable (e.g. `[200, 201, 204]`).
13. Add support for `has_header` assertion. Parameters: `name` (string). Validates that the response includes a header with the given name, regardless of its value. Fails if the header is absent.
14. Add support for `value_type` assertion. Parameters: `path`, `expected` (one of: `string`, `number`, `boolean`, `array`, `object`, `null`). Validates that the JSON value at the given path is of the specified type.
15. Add support for `greater_than` assertion. Parameters: `path`, `expected` (number). Validates that the numeric value at the given path is strictly greater than `expected`. If the value is a string, attempt to parse it as a number first. Fails if the value is not numeric or parsing fails.
16. Add support for `less_than` assertion. Parameters: `path`, `expected` (number). Validates that the numeric value at the given path is strictly less than `expected`. If the value is a string, attempt to parse it as a number first. Fails if the value is not numeric or parsing fails.
17. Add support for `greater_than_or_equal` assertion. Parameters: `path`, `expected` (number). Validates that the numeric value at the given path is greater than or equal to `expected`. If the value is a string, attempt to parse it as a number first. Fails if the value is not numeric or parsing fails.
18. Add support for `less_than_or_equal` assertion. Parameters: `path`, `expected` (number). Validates that the numeric value at the given path is less than or equal to `expected`. If the value is a string, attempt to parse it as a number first. Fails if the value is not numeric or parsing fails.
19. Add support for `array_size` assertion. Parameters: `path`, `expected` (integer). Validates that the array at the given path has exactly `expected` number of elements. Fails if the value at path is not an array.
20. Add support for `array_size_min` assertion. Parameters: `path`, `min` (integer). Validates that the array at the given path has at least `min` elements. Fails if the value at path is not an array.
21. Add support for `array_size_max` assertion. Parameters: `path`, `max` (integer). Validates that the array at the given path has at most `max` elements. Fails if the value at path is not an array.
22. Add support for `array_contains` assertion. Parameters: `path`, `expected` (scalar value — string, number, or boolean). Validates that the array at the given path contains the expected value. Fails if the value at path is not an array or does not contain the expected item.
23. Add support for `array_contains_all` assertion. Parameters: `path`, `expected` (array of scalar values). Validates that the array at the given path contains all values listed in `expected`. Fails if the value at path is not an array or is missing any of the expected items.
24. Add support for `array_is_empty` assertion. Parameters: `path`. Validates that the array at the given path exists and has zero elements. Fails if the value at path is not an array or has one or more elements.
25. Add support for `array_is_not_empty` assertion. Parameters: `path`. Validates that the array at the given path exists and has at least one element. Fails if the value at path is not an array or has zero elements.
